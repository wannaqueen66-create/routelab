# 天气概览增强与路线天气留档设计

日期：2026-03-27

## 1. 背景与目标

当前项目已有两类天气展示：

- 首页“今日天气与建议”，用于展示用户当前位置的实时天气概览。
- 路线详情页“出行天气”，用于展示某次路线记录关联的天气快照。

现状问题：

1. 首页天气概览字段不完整，缺少风向与更明确的空气质量展示。
2. 首页只能看当前快照，无法查看前后时间窗口内的天气变化。
3. 路线详情页虽然已有天气快照，但展示项与首页概览不一致，且“当次出行天气留档”的定位还不够明确。
4. 后端当前 `/weather` 接口只返回当前天气快照，不足以支撑 24 小时变化图。

本次设计目标：

1. 增强首页天气概览，补充风向与空气质量展示。
2. 在首页提供“按指标点击查看前 12 小时 + 未来 12 小时变化图”的能力。
3. 明确路线详情页承担“当次出行天气留档”，并继续将天气快照存入数据库。
4. 保持范围可控：本轮不做独立天气页，不做路线详情图表，不做污染物多曲线复杂展示。

## 2. 范围与非目标

### 2.1 本次范围

#### 首页天气卡

- 保留现有顶部天气概览结构。
- 将指标区整理为 4 个可点击指标：
  - 温度 / 体感
  - 湿度
  - 风力 / 风向
  - 空气 / 污染
- 默认不显示 24 小时详情区。
- 点击某个指标后，在天气卡内部展开对应详情图。
- 再次点击当前已选指标时，收起详情区。
- 点击其他指标时，在同一详情区切换图表内容。

#### 路线详情页天气卡

- 明确该模块是“当次出行天气留档”。
- 继续使用数据库中 `routes.weather` 存储的天气快照。
- 展示与首页概览同级别的信息：
  - 温度
  - 天气文字
  - 体感温度
  - 湿度
  - 风力 + 风向
  - 空气质量（AQI + 等级）
  - 运动建议
- 不展示污染物拆解细项。
- 不展示 24 小时图表。

#### 后端天气接口

- 在现有实时天气查询能力上，新增小时级天气时间线数据。
- 提供前 12 小时 + 未来 12 小时的统一 hourly 数据供首页图表使用。
- 路线详情页继续使用快照数据，不依赖 hourly 数据。

### 2.2 非目标

本次明确不做：

- 独立天气详情页
- 首页以外页面的小时级图表
- 路线详情页的天气变化图
- 污染物多条折线同时展示
- 首页天气卡的复杂手势交互
- 将首页实时 hourly 数据写入路线记录

## 3. 现有系统认知

### 3.1 首页天气展示现状

首页天气卡位于 `pages/index/index.wxml`，当前已展示：

- 温度
- 天气文字
- 体感温度
- 城市
- 运动建议
- 湿度
- 风力
- 空气质量

数据获取与格式化流程：

- `pages/index/index.js`
  - 通过 `api.getWeatherSnapshotSafe({ latitude, longitude })` 获取天气数据。
  - 通过 `formatWeatherPayload(payload)` 格式化为 UI 状态。
- `services/weather-local.js`
  - 负责将后端 payload 归一化为前端展示格式。

### 3.2 路线天气留档现状

路线创建后，后端会异步回填天气快照：

- `cloud/server/src/routes/routes.js`
  - `backfillWeatherForRoute(routeId, points)` 会拉取天气与空气质量并写回路线。
- `cloud/server/src/models/routeModel.js`
  - `createRoute` 支持写入 `weather` 字段。
  - `updateRouteWeather(routeId, weather)` 会在 `weather IS NULL` 时回填天气快照。

结论：当前系统已经具备“路线天气入库”的基础，本次不需要额外引入新的路线天气持久化机制。

## 4. 产品设计

## 4.1 首页天气卡最终结构

首页天气卡由两层组成：

### 第一层：天气概览区

保留现有大体结构：

- 当前温度
- 天气现象文字
- 体感温度
- 城市 / 位置
- 一句运动建议

该区域仍是卡片顶部的总览，不承载图表切换交互。

### 第二层：可点击指标区

指标区统一改为 4 个可点击项：

1. **温度 / 体感**
   - 概览值：当前温度 + 当前体感
2. **湿度**
   - 概览值：当前湿度
3. **风力 / 风向**
   - 概览值：当前风速 + 当前风向文字
4. **空气 / 污染**
   - 概览值：当前 AQI + 空气质量等级

设计原则：

- 指标数量与后续详情视图一一对应，保证用户点击前后心智一致。
- “空气 / 污染”在概览层只展示 1 个入口，不拆成多个污染物入口，避免天气卡过重。

## 4.2 首页天气详情交互

### 初始状态

- 用户首次进入首页时，仅显示概览区与指标区。
- 不显示 24 小时详情区。

### 展开逻辑

- 点击任一指标：
  - 在当前天气卡内部展开详情区。
  - 默认滚动不跳页、不打开弹层。
- 再次点击当前已选指标：
  - 收起详情区。
- 点击其他指标：
  - 保持详情区打开，仅切换图表与摘要内容。

### 详情区位置

详情区位于指标区下方，仍属于同一张天气卡的一部分。

这样可以满足：

- 首页保持轻量
- 详情只在用户有意图时出现
- 不需要新页面与新导航

## 4.3 首页详情区内容定义

详情区统一展示最近 24 小时窗口：

- 前 12 小时
- 当前时刻
- 未来 12 小时

### 温度 / 体感

展示内容：

- 主图：温度折线
- 辅助线：体感温度折线
- 摘要：最低 / 当前 / 最高

### 湿度

展示内容：

- 主图：湿度折线
- 摘要：最低 / 当前 / 最高

### 风力 / 风向

展示内容：

- 主图：风速折线
- 文本摘要：
  - 当前风向
  - 主导风向
  - 最大风速时段

说明：

- 风向不单独绘制成复杂图形，本轮先做文本摘要，降低实现与阅读成本。

### 空气 / 污染

展示内容：

- 主图：AQI 折线
- 文本摘要：
  - 当前 AQI
  - 空气质量等级
  - 若后端可稳定提供，则可补充当前 PM2.5 / PM10 / O3 作为辅助文案

说明：

- 本轮不绘制 PM2.5 / PM10 / O3 / SO2 / NO2 / CO 多条线。
- 本轮空气详情的核心仍然是 AQI 的时间变化。

## 4.4 路线详情页天气卡设计

路线详情页天气模块继续定位为“当次出行天气留档”。

### 展示项

建议展示：

- 温度
- 天气文字
- 体感
- 湿度
- 风力
- 风向
- 空气质量（AQI + 等级）
- 建议

### 不展示项

- PM2.5 / PM10 / O3 等污染物拆解
- 24 小时变化图
- 可点击切换的指标卡

### 理由

路线详情页的目标是“回看这次路线当时的天气情况”，不是做实时天气分析。

因此：

- 保留快照式展示最合适。
- 与首页概览字段保持一致即可。
- 这样用户在两个场景下看到的是同一套天气语言，但职责不同：
  - 首页：实时分析
  - 路线详情：历史留档

## 5. 数据设计

## 5.1 路线天气快照

继续沿用 `routes.weather` 存储 JSON 快照。

推荐快照字段最小集合：

```json
{
  "temperature": 18.6,
  "apparentTemperature": 17.9,
  "weatherCode": 3,
  "weatherText": "多云",
  "humidity": 62,
  "windSpeed": 3.4,
  "windDirection": 45,
  "windDirectionText": "东北风",
  "aqi": 58,
  "airLevel": "Moderate",
  "airCategory": "moderate",
  "suggestion": "天气不错，可以按计划进行户外运动。",
  "source": "qweather",
  "fetchedAt": 1774540000000
}
```

说明：

- 即使数据库里保留更多字段，路线详情页 UI 也只消费概览级字段。
- 本轮不要求修改路由天气快照为 hourly 结构。

## 5.2 首页 hourly 数据

首页需要新的 hourly 数据结构，用于图表与摘要。

建议接口返回结构：

```json
{
  "current": {
    "temperature": 18.6,
    "apparentTemperature": 17.9,
    "weatherText": "多云",
    "humidity": 62,
    "windSpeed": 3.4,
    "windDirection": 45,
    "windDirectionText": "东北风",
    "aqi": 58,
    "airLevel": "Moderate",
    "airCategory": "moderate",
    "suggestion": "天气不错，可以按计划进行户外运动。",
    "source": "qweather",
    "fetchedAt": 1774540000000
  },
  "hourly": [
    {
      "time": 1774496800000,
      "temperature": 14.2,
      "apparentTemperature": 13.8,
      "humidity": 71,
      "windSpeed": 2.1,
      "windDirection": 20,
      "windDirectionText": "东北风",
      "aqi": 44,
      "pm25": 18,
      "pm10": 29,
      "o3": 71,
      "so2": null,
      "no2": null,
      "co": null
    }
  ]
}
```

说明：

- `current` 用于首页顶部概览与默认当前值。
- `hourly` 用于详情图与指标摘要。
- 允许某些污染物字段为空；前端应优雅降级。

## 5.3 数据持久化边界

### 会入库的数据

- 路线详情页使用的天气快照继续入库，写入 `routes.weather`。

### 不入库的数据

- 首页 24 小时 hourly 数据不写入路线记录。
- 首页 hourly 数据只做实时查询与前端短时缓存。

理由：

- 首页 hourly 数据具有明显时效性。
- 它的职责是支持实时查看，不是历史归档。
- 如果将其与路线快照混存，会导致数据体量扩大、语义混乱。

## 6. 接口设计

## 6.1 首页天气接口策略

建议在现有天气接口能力基础上扩展，而不是让前端拼多个来源。

推荐方式：

- 保留现有天气快照能力。
- 新增一个面向首页的 hourly 接口，或扩展现有 `/weather` 响应以包含 hourly 数据。

推荐响应原则：

- 首页前端一次请求拿到当前概览与 hourly 时间线。
- 后端负责对不同 provider 的字段进行归一化。
- 前端不感知 provider 细节。

若选扩展现有 `/weather`：

- 优点：前端调用点少
- 缺点：接口职责从“快照”变成“快照 + 时间线”，语义变重

若新增独立 hourly 接口：

- 优点：职责清晰，便于缓存与演进
- 缺点：前端需多一次请求或后续封装

本次推荐：

**新增独立的 hourly 天气接口**，由首页天气模块内部统一封装调用。

原因：

- 路由详情页仍然只需要快照。
- 首页图表是新能力，独立接口边界更清晰。
- 后续如果要增加日视图或更长趋势，更容易扩展。

## 7. 前端状态设计

首页天气状态建议拆成两层：

### 概览层状态

负责：

- 当前天气展示
- 加载 / 错误态
- 当前城市

### 详情层状态

负责：

- 当前选中的指标 key
- 详情区是否展开
- hourly 数据
- 当前图表渲染数据

建议状态字段示例：

```js
{
  weather: {
    loading: false,
    ready: true,
    current: { ... },
    hourly: [],
    activeMetric: '',
    detailVisible: false,
    error: ''
  }
}
```

说明：

- `activeMetric` 为空表示默认隐藏。
- `detailVisible` 可以由 `activeMetric` 推导，也可以显式保存。
- 图表组件接收标准化数据，不直接理解原始接口结构。

## 8. UI 组件拆分建议

为了降低首页复杂度，建议将天气模块拆为三个子职责：

1. **WeatherSummaryCard**
   - 顶部总览
   - 当前建议
2. **WeatherMetricTabs**
   - 四个可点击指标
   - 选中态 / 收起态逻辑
3. **WeatherMetricDetail**
   - 根据 activeMetric 渲染图表与摘要

即使仍在小程序页面文件中实现，也应按这三个职责组织代码或抽离辅助函数，避免首页页面脚本进一步膨胀。

## 9. 图表方案

本次设计不强绑定具体图表库，但要求满足：

- 能绘制单线折线图
- 温度支持双线（温度 + 体感）
- 支持移动端小程序环境
- 支持较轻量的样式定制

如果现有项目没有图表依赖，优先选择：

- 小程序兼容、集成成本低的轻量图表方案
- 或者自绘简单折线图（若只需基础能力）

本轮要求的图表能力有限，避免为了这一处天气模块引入过重的图表系统。

## 10. 错误处理与降级

### 首页概览

- 如果当前天气快照获取失败，继续沿用现有错误提示策略。

### 首页 hourly 详情

- 如果 hourly 获取失败：
  - 概览仍可正常显示。
  - 点击指标后显示“暂无 24 小时天气变化数据”。
  - 不影响首页其它内容。

### 空气质量明细缺失

- 若某些污染物字段为空，只展示 AQI 与等级。
- 不因为污染物缺失而整体隐藏“空气 / 污染”指标。

### 风向缺失

- 若风向角度缺失但文字存在，优先显示文字。
- 若文字也缺失，则仅显示风速。

## 11. 测试要求

### 前端

至少验证：

1. 首页首次进入时详情区默认隐藏。
2. 点击任一指标后，详情区正确展开。
3. 再次点击当前指标后，详情区正确收起。
4. 切换指标时，图表内容正确切换。
5. 空气指标在只有 AQI、没有污染物细项时仍可正常展示。
6. 路线详情页正确显示风向与空气质量概览。

### 后端

至少验证：

1. 路线 weather 快照继续正常写入数据库。
2. 首页 hourly 接口在 provider 正常时返回完整 24 小时窗口。
3. provider 部分字段缺失时，返回结构仍满足前端消费。
4. hourly 获取失败时，不影响快照接口与路线天气回填。

## 12. 实施顺序建议

建议按以下顺序实施：

1. 扩展后端 hourly 天气查询能力与接口。
2. 扩展首页天气数据服务层，接入 hourly 数据。
3. 改造首页天气卡为“4 指标 + 卡内展开详情”。
4. 在路线详情页补齐风向与空气质量概览展示。
5. 补充验证与回归测试。

## 13. 最终决策摘要

本次设计的最终决策如下：

1. 首页天气卡增强为 4 个可点击指标：温度 / 体感、湿度、风力 / 风向、空气 / 污染。
2. 24 小时详情区仅放在首页，且采用卡片内展开。
3. 首页首次进入时详情区默认隐藏。
4. 空气详情以 AQI 变化为主，不展示多污染物曲线。
5. 路线详情页继续承担“当次出行天气留档”。
6. 路线天气继续入库，沿用 `routes.weather` 快照存储。
7. 路线详情页只展示首页概览级别的天气信息，不额外展开污染物细项。
8. 首页 hourly 数据不写入路线记录，只作为实时分析数据使用。
