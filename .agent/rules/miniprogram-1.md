## RouteLab 项目协作规则（Always On）

### 适用范围
- 本规则适用于 **RouteLab** 仓库内所有改动与协作（代码、配置、文档）。
- 重点约束：`cloud/`（云端部署相关）目录的开发与验证方式、以及文档同步更新要求。

---

### 规则 1：`cloud/` 目录只面向云端部署，不在本机做测试验证
- `cloud/` 文件夹是 **后续放到云端服务器部署** 的交付内容。
- 云端目标系统为 **Ubuntu**，因此：
  - **不需要**在我的本地电脑对 `cloud/` 目录进行运行/联调/容器启动等测试。
  - 任何涉及 `cloud/` 的修改，应以“可在 Ubuntu 环境部署成功”为目标进行设计与实现，而不是以本地环境可运行作为前提。
- 本地侧允许做的验证（如确有必要）：
  - 仅限静态检查与基本构建层面的确认（例如语法/依赖合理性、配置不明显冲突等），但**不要求也不进行**完整部署链路验证。

---

### 规则 2：修改项目后必须同步更新文档（中文）
每次对项目做出有效修改（功能、接口、配置、目录结构、部署方式、依赖变化等）后，必须**及时**维护以下文档，并且**全部使用中文**：

> “及时”默认指：**同一次提交/PR 中同步更新**（或至少在同一天内补齐）。

1) **更新 `README.md`（项目说明/使用入口）**
- 变更点涉及的功能说明、模块结构、启动方式、开发提示等内容要同步更新。
- 保持 README 的信息能让新读者快速理解：项目是什么、包含哪些端、如何使用与运行。

2) **更新 `deployment.md`（`cloud/` 云端部署指南）**
- 任何影响云端部署的改动都必须写入 `deployment.md`，包括但不限于：
  - `cloud/` 目录内代码与配置的变更
  - Docker / Docker Compose / Nginx / 数据库（PostgreSQL）相关配置变化
  - 环境变量、端口、反向代理路由、构建与启动命令
  - 升级/迁移步骤、常见错误与排查方式
- 目标：仅凭 `deployment.md`，在一台干净的 Ubuntu 服务器上可以按步骤完成部署（不依赖口头说明）。
